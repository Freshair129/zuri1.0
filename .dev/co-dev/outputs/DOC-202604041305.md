## CTO Review

**Spec:** FEAT18 — Express Accounting Integration
**Decision:** ADR_REQUIRED + NEEDS_REVISION

---

### Checks

- [x] Data flow follows QStash worker pattern (UI → event → QStash → Express API) — correct
- [x] No direct Express API calls from UI routes — spec uses QStash workers
- [x] `tenant_id` on all 3 new tables — correct
- [x] NFR3 QStash retry ≥5 — mentioned for network timeout
- [ ] **API key encryption strategy — not defined** (just says "encrypted" but no HOW)
- [ ] **RBAC — not defined** (who can configure? who can trigger manual sync?)
- [ ] **Redis cache for sync log dashboard — not mentioned** (NFR2)
- [ ] **No ADR written yet** — spec itself acknowledges 2 ADRs needed but hasn't created them

---

### Issues Found

**1. Schema — 3 new tables require ADR (G-DEV-04)**
`integration_configs`, `integration_sync_logs`, `integration_document_refs` are entirely new tables. Per our rules, schema changes need an ADR + migration plan before code.

**2. API Key Encryption — no strategy**
Spec says `api_key_encrypted` but doesn't specify:
- Encryption method (AES-256-GCM? Vercel encrypted env + per-tenant key?)
- Key management (where is the encryption key stored?)
- This is a security-critical decision → **ADR required**

**3. RBAC missing entirely**
Who can:
- Configure Express connection? (likely OWNER, MANAGER, FINANCE only)
- Trigger manual sync? (OWNER, MANAGER, FINANCE?)
- View sync logs? (add FINANCE role)
- The spec must map actions to `can(roles, domain, action)` from `src/lib/permissionMatrix.js`

**4. New external dependency — Express REST API**
First external accounting API integration in Zuri. This sets the **pattern** for all future integrations (FlowAccount, PEAK, Sage mentioned in Phase 3). Must be designed as a generic integration layer, not Express-specific. **ADR required.**

**5. Real-time sync mode — event sourcing gap**
Spec says "order `paid` event" triggers real-time sync, but Zuri currently has no event bus. How does the POS `paid` action trigger a QStash publish? Options:
- API route publishes to QStash after order update (adds latency to POS flow)
- DB trigger via Supabase webhook (new infra)
- Repository hook pattern (new pattern)
This is an architectural decision → belongs in the sync strategy ADR.

**6. `sync_options_json` / `account_mapping_json` — JSON blobs in Postgres**
These should be typed. If we need to query by sync option or account mapping, JSON blobs become a pain. Consider:
- `integration_account_mappings` as a separate table (category → Express account code)
- Or at minimum, use Prisma's `Json` type with documented structure

**7. Idempotency key — good but incomplete**
`ZRI-{order_id}` for dedup is good. But what about:
- Customer sync? (what's the idempotency key — `customer_id`?)
- Expense sync? (what's the reference — `expense_id`?)
- Spec only covers invoice dedup.

**8. Open questions block implementation**
4 open questions in Section 10 — especially "Express API: ต้อง apply เป็น partner?" is a **blocker**. If Express requires partner API access, the entire timeline shifts.

---

### ADRs Required (3)

| ADR | Decision |
|-----|----------|
| **ADR-069: External Integration Architecture** | Generic integration layer design — provider interface, config storage, sync worker pattern. Sets pattern for Express + future providers. |
| **ADR-070: API Key Encryption at Rest** | How tenant API keys are encrypted, where master keys live, rotation strategy. |
| **ADR-071: Express Sync Strategy** | Real-time event trigger mechanism, batch cron design, idempotency across all entity types. |

---

### Notes for Implementation (after ADRs approved)

1. **Design a `ProviderAdapter` interface** — `syncInvoice()`, `syncCustomer()`, `syncExpense()` — so Phase 3 providers (FlowAccount, PEAK) just implement the adapter
2. **New repository:** `src/lib/repositories/integrationRepo.js` — all 3 tables
3. **New worker:** `/api/workers/sync-express` — QStash target with `maxDuration: 300` in `vercel.json`
4. **Settings page** needs `can(roles, 'integration', 'configure')` guard — add to `permissionMatrix.js`
5. **Sync log page** should use Redis cache (NFR2) since it's a dashboard-style view
6. Gotchas to watch: G-WH-03 (maxDuration), G-DB-03 ($transaction for config upsert), G-DEV-04 (ADR before code)

---

**Bottom line:** Good spec foundation — data flow is correct, QStash pattern is right, idempotency thinking is there. But this is Zuri's **first external integration** and sets the pattern for all future ones. Needs ADRs before any code touches `schema.prisma`. Resolve the 4 open questions (especially Express partner API access) first.
