# Feature Spec: CRM Diagnostic & Simulation Tool (Test CRM)

**Version:** 1.0.0
**Date:** Saturday, April 4, 2026
**Status:** DRAFT
**Author:** PM Agent

---

## 1. Overview
The **Test CRM** feature is a diagnostic and simulation suite designed to ensure the integrity of the Zuri CDP layer. It allows developers and managers to validate identity resolution logic, detect data anomalies (duplicates/orphans), and simulate lead ingestion flows to verify AI-driven intent analysis without affecting production metrics.

## 2. User Stories
- As a **DEV**, I want to simulate Facebook and LINE webhook payloads, so that I can verify that identity merging (linking social IDs to phone numbers) works correctly.
- As a **MANAGER**, I want to run a CRM health audit, so that I can identify and merge duplicate customer records that were missed by the automated system.
- As a **SALES** agent, I want to input a "test" phone number and see how the system would resolve its identity, so that I can troubleshoot why a customer is seeing multiple profiles.

## 3. Acceptance Criteria
- [ ] Ability to simulate a `MESSAGE` webhook event for both Facebook and LINE channels.
- [ ] Diagnostic report showing potential duplicates based on fuzzy name matching and normalized phone numbers.
- [ ] Validation tool to check if all `Customer` records in a tenant have a valid E.164 `phone_primary`.
- [ ] Trace log for identity resolution: Show step-by-step why record A was merged into record B.
- [ ] "Dry-run" mode for lead ingestion that calculates `intentScore` and `lifecycleStage` without writing to the `Customer` table.

## 4. Data Flow
1. **Trigger:** User selects "Simulate Webhook" in the CRM Test Panel.
2. **Action:** UI sends a mock payload (JSON) to `/api/crm/test/simulate`.
3. **Logic:** The `identityService.js` processes the payload (dry-run flag = true).
4. **Validation:** System checks `tenant_id` isolation and performs phone normalization via `normalizePhone()`.
5. **Output:** API returns a `SimulationResult` object containing the predicted `customerId` and any merge conflicts found.
6. **Persistence:** Simulation results are stored in `CrmDiagnosticLog` for audit trail.

## 5. API Endpoints Required
| Method | Path | Auth | Roles | Description |
|--------|------|------|-------|-------------|
| POST   | /api/crm/test/simulate | JWT | DEV, OWNER | Simulates lead ingestion and identity resolution |
| GET    | /api/crm/test/audit | JWT | DEV, MGR | Runs a health scan on all CRM records for the tenant |
| POST   | /api/crm/test/merge-fix | JWT | DEV, MGR | Manually merges records identified in the audit |

## 6. Database Changes
- New tables:
  - `CrmDiagnosticLog`: Stores history of simulations and audit results (`id`, `tenantId`, `type`, `payload`, `result`, `createdAt`).
- Modified tables:
  - `Customer`: Add `is_test` (Boolean, default: false) to flag simulation data if users choose to persist it.
- New indexes:
  - `@@index([tenantId, createdAt])` on `CrmDiagnosticLog`.

## 7. Roles & Permissions
| Action | Allowed Roles |
|--------|--------------|
| Run CRM Audit | DEV, OWNER, MANAGER |
| Simulate Webhooks | DEV |
| Manual Merge Records | DEV, OWNER, MANAGER |
| Delete Test Data | DEV |

## 8. UI Components Required
- `CrmTestPanel` — Central diagnostic hub located in `src/components/crm/CrmTestPanel.js`.
- `IdentityTraceView` — Visual step-by-step flow of merge logic in `src/components/crm/IdentityTraceView.js`.
- `HealthAuditDashboard` — Summary charts of CRM data quality in `src/app/(dashboard)/crm/health/page.jsx`.

## 9. Tech Notes & Gotchas
- **G-DB-01 (Phone Merge):** Simulation must strictly follow the E.164 normalization rule to prevent creating fragmented test data.
- **G-MT-01 (Tenant Leak):** Audit queries MUST include `WHERE tenant_id = current_tenant` to ensure one school cannot see another's diagnostic logs.
- **NFR1:** Webhook simulation endpoint should mimic the < 200ms response time by using a mock QStash trigger.

## 10. Out of Scope
- Fixes for external Meta/LINE API data (only internal DB records are audited).
- Real-time cleaning of incoming production webhooks (this is a diagnostic tool, not the primary worker).
- Testing of POS or Kitchen Ops modules (restricted to CRM/Inbox identity).
