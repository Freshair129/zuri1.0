## CTO Review

**Spec:** FEAT18 — Express Accounting Integration
**Decision:** ADR_REQUIRED + NEEDS_REVISION

### Checks
- [x] Data flow follows: UI → API → Repo → DB → QStash Worker → Express API ✅
- [x] No direct Meta/LINE API calls from routes ✅ (not applicable here)
- [x] tenantId isolation in every query — spec includes `tenant_id` in all 3 new tables ✅
- [x] Webhook returns 200 immediately — Phase 2 concern, not Phase 1 ✅
- [x] Dashboard routes have Redis cache — sync log page needs this (see below) ⚠️
- [x] Workers throw error for QStash retry (NFR3) — mentioned ✅
- [ ] Identity upsert in transaction — **not addressed** for customer sync ❌
- [x] No new large dependencies without ADR — Express API client is new external dep ✅
- [ ] Schema changes without ADR — **3 new tables, need ADR** ❌

### Issues Found

**1. Missing ADR — must write before implementation**
- 3 new DB tables (`integration_configs`, `integration_sync_logs`, `integration_document_refs`) = significant schema change → ADR required per G-DEV-04
- API key encryption at rest — spec mentions `api_key_encrypted` but no strategy (KMS? Vercel env? AES-256 with what key management?)
- The spec itself correctly identifies 2 ADRs needed (§11) but they have `ADR-XXX` placeholders — **these must be written and numbered before coding starts**

**2. API Key Security — underspecified (CRITICAL)**
- `api_key_encrypted` column exists but no encryption/decryption strategy
- On Vercel serverless, there's no persistent memory for key management
- Options: (a) encrypt with `ENCRYPTION_KEY` from Vercel env var + AES-256-GCM, (b) Supabase Vault, (c) store in Vercel env per tenant (doesn't scale)
- **Recommendation:** AES-256-GCM with key from `process.env.ENCRYPTION_KEY`, decrypt at worker runtime only. Document in ADR.

**3. Customer sync needs `$transaction` (NFR5 / G-DB-03)**
- §5.2 syncs CRM Customer → Express. If the Express API call fails mid-batch, what's the rollback strategy?
- The `integration_document_refs` status tracking helps, but the spec doesn't mention wrapping the "lookup customer + create Express contact + save ref" in a transaction
- **Add:** Each sync item should be wrapped: write `document_ref` with status `pending` → call Express API → update to `synced`/`failed`, all in a try/catch per item

**4. QStash free tier concern (G-WH-06)**
- Real-time mode fires a QStash message per `order.paid` event. A busy school with 50+ orders/day across multiple tenants could hit the 500 msg/day free tier limit fast
- **Recommendation:** Default to daily batch mode. Document real-time mode as requiring Upstash paid plan.

**5. Sync log page missing Redis cache (NFR2)**
- §7.1 shows a sync log UI table — this will be a dashboard-style page
- No mention of Redis cache for the sync log API endpoint
- **Add:** `getOrSet('express-sync-log:{tenantId}', fn, 300)` per NFR2

**6. Idempotency gap — expense sync**
- §7.3 covers idempotency for orders via `ZRI-{order_id}` — good
- But expense sync (§5.3) has no equivalent reference key specified
- **Add:** `ZRI-EXP-{expense_id}` as the Express document reference for expenses

**7. Repository pattern not specified**
- Spec doesn't mention which repository files will be created
- **Need:** `src/lib/repositories/integrationRepository.js` — all Express DB operations must go through this, not direct Prisma calls from workers

**8. RBAC — who can configure?**
- §6 shows a settings page but doesn't specify which roles can access it
- **Recommendation:** OWNER + MANAGER + FINANCE only (API keys are sensitive). Use `can(roles, 'integration', 'manage')` — needs a new permission domain in `permissionMatrix.js`

### ADR Required

1. **ADR-069: Express Integration Schema & Sync Strategy** — covers 3 new tables, real-time vs batch trade-off, QStash message volume, idempotency keys
2. **ADR-070: Integration API Key Encryption at Rest** — encryption algorithm, key management on Vercel serverless, rotation strategy
