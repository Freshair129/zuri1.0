# ADR-069: AI Context Layer (NotebookLM)

## Status
PROPOSED

## Context
As the Zuri Platform evolves into an "AI-First" business engine, the requirement for high-density, domain-specific, and tenant-scoped context injection has grown. Standard RAG (Retrieval Augmented Generation) often lacks the "holistic" understanding of a business's operational style, product catalog, and historical context needed for high-quality responses (e.g. `DSB v2`, `Compose Reply`).

## Decision
We will implement an **AI Context Layer** using **NotebookLM-style "Notebooks"** as the primary unit of context management.

1.  **Logical Mapping:** Each Tenant will have its own set of Notebooks (e.g. `{tenant}-chat-intelligence`, `{tenant}-pdad-framework`).
2.  **Integration Vector:** We will use the **NotebookLM API** (or equivalent high-context window injection via Gemini 1.5 Pro/2.0 Flash) to feed these notebooks into every AI interaction.
3.  **Source Material:**
    *   **Platform Docs:** PRD, ERD, ADRs (from `docs/`).
    *   **Tenant Data:** Product catalogs, historical chat logs (anonymized), business guidelines.
    *   **Domain logic:** PDAD framework rules.
4.  **Auto-Synchronization:** Use a QStash worker to periodically update Notebook content when significant changes occur in the tenant's data (e.g. new product launched).
5.  **Access Control:** All Notebooks are read-only for the AI Agents. No direct user access to raw Notebook content; it is exposed only through the "Ask AI" or "Assist" features.

## Rationale
- **High Recall:** NotebookLM (via Gemini's massive context window) enables the AI to "remember" the entire business context rather than just a few relevant snippets.
- **Style Extraction:** Enables A2 (Agent Mode) by allowing the AI to analyze hundreds of past conversations in one context window to extract a specific admin's tone.
- **Consistency:** Ensures the AI follows the latest PRD/ADR guidelines by treating documentation as the Single Source of Truth (SSOT).

## Impact
- **Cost:** Requires Gemini Pro/Flash API calls. Notebook creation/sync is an async cost.
- **Latency:** Slightly higher initial "cold start" for context-heavy requests (mitigated by Gemini's context caching).
- **Complexity:** Adds a new layer to the `core/ai` module for managing Notebook IDs and Synchronization status.

## Alternatives Considered
- **Vector DB (Pinecone/Supabase Vector):** Better for millions of small items, but worse for "understanding" complex relationships in technical docs or brand guidelines.
- **Direct Prompting:** Limited by token windows and degrades performance as context Grows.
