## Color Philosophy: 60-30-10 Rule

- **60% — Neutrals**: `#F5F8FB` (cool white bg) + `#FFF8F0` (warm white surface)
- **30% — Secondary**: `#101828` (deep navy) + `#1A1A1A` (off-black)
- **10% — Accent**: `#D4A017` (saffron gold) — CTAs and key actions ONLY
- **Destructive**: `#DC2626` — always red, never gold
- **Principle**: No pure colors. Every neutral has a tint (blue or warm). Gold is reserved — overuse kills its impact.

## Origin of the Name

**Zuri** (ซูริ) is inspired by Suri, a character from the Thai indie film *Mary Is Happy, Mary Is Happy* (2013, directed by Nawapol Thamrongrattanarit / นวพล ธำรงรัตนฤทธิ์). The name captures a spirit that is youthful, observant, quietly dependable, and unmistakably Thai.

**Tagline (EN):** Your business's best friend
**Tagline (TH):** มีซูริ ไม่ต้องเป็นห่วง

### 1.3 Typography

#### Font Stack

| Role | Font | Google Fonts Link | Fallback |
|---|---|---|---|
| **Heading** | **IBM Plex Sans Thai** (700, 600) | `IBM+Plex+Sans+Thai:wght@400;500;600;700` | `Noto Sans Thai`, `sans-serif` |
| **Body** | **IBM Plex Sans Thai** (400, 500) | same as above | `Noto Sans Thai`, `sans-serif` |
| **UI Labels** | **Prompt** (400, 500, 600) | `Prompt:wght@400;500;600` | `IBM Plex Sans Thai`, `sans-serif` |
| **Mono / Data** | **IBM Plex Mono** (400, 500) | `IBM+Plex+Mono:wght@400;500` | `Fira Code`, `monospace` |

**Why IBM Plex Sans Thai:** Excellent Thai glyph support, clean geometry, professional yet approachable. Designed for digital interfaces. Free, open-source.

**Why Prompt as alternative for UI:** Slightly more compact than IBM Plex, good for dense UI elements like table headers, badges, and navigation labels.

#### Type Scale

Based on a 1.250 ratio (Major Third), base size 16px.

| Token | Size (px) | Size (rem) | Line Height | Weight | Usage |
|---|---|---|---|---|---|
| `text-xs` | 12 | 0.75 | 1.5 (18px) | 400 | Captions, timestamps |
| `text-sm` | 14 | 0.875 | 1.5 (21px) | 400 | Secondary text, table cells |
| `text-base` | 16 | 1.0 | 1.625 (26px) | 400 | Body text, chat messages |
| `text-lg` | 18 | 1.125 | 1.555 (28px) | 500 | Emphasized body, card titles |
| `text-xl` | 20 | 1.25 | 1.4 (28px) | 600 | Section headings |
| `text-2xl` | 24 | 1.5 | 1.333 (32px) | 600 | Page headings |
| `text-3xl` | 30 | 1.875 | 1.267 (38px) | 700 | Dashboard metrics |
| `text-4xl` | 36 | 2.25 | 1.222 (44px) | 700 | Hero heading (landing page) |

**Thai text note:** Thai script has taller ascenders/descenders than Latin. Always use `line-height >= 1.5` for body text to prevent clipping of Thai tone marks (สระบน/สระล่าง).

## 2. UI/UX Principles

### 2.1 Design Philosophy

1. **Mobile-first, desktop-optimized.** Thai SME staff (receptionists, kitchen teams, sales agents) primarily use phones during the day. Desktop is for managers reviewing dashboards and doing data entry.

2. **Familiar patterns.** Zuri should feel as intuitive as LINE or Facebook — apps Thai users already know. No learning curve for basic tasks.

3. **Progressive disclosure.** Show only what is needed now. Advanced features reveal themselves as the user grows.

4. **Thai-optimized density.** Thai text is inherently denser than English. Allow generous line height and padding. Never cram Thai text into tight spaces.

5. **Speed is a feature.** Skeleton screens, optimistic updates, fast transitions. The app should feel instant.

---

### 2.1.1 Typography Pairing Rule

**Pair Serif + Sans-serif** for readability. All-sans-serif interfaces feel flat and harder to scan after extended use.

| Role | Category | Font | Rationale |
|---|---|---|---|
| **Display / Hero** | Sans-serif (ไม่มีหัว) | IBM Plex Sans Thai 700 | Large sizes read well without serifs |
| **Section headings** | Sans-serif (ไม่มีหัว) | IBM Plex Sans Thai 600 | Clean hierarchy |
| **Body text / Long copy** | Serif (มีหัว) | Lora or Noto Serif Thai 400 | Serifs guide the eye — reduces fatigue on paragraphs |
| **UI labels / Navigation** | Sans-serif (ไม่มีหัว) | Prompt 500 | Compact, crisp in tight UI density |
| **Data / Numbers** | Monospace | IBM Plex Mono 400 | Tabular figures align in columns |

> **Rule:** Never use a single font family across the entire UI. Contrast between serif body and sans-serif headings creates visual rhythm and reduces reading fatigue — especially in Thai where long vowels and tone marks compete for vertical space.

---

### 2.1.2 Color Saturation by Mode

| Mode | Saturation | Why |
|---|---|---|
| **Light mode** | **100%** | Full vibrancy — screen reflects ambient light naturally |
| **Dark mode** | **70%** | Reduce saturation 30% — oversaturated colors on dark bg cause eye strain and look "neon" |

Apply by reducing CSS `saturation` filter or defining separate dark-mode color tokens at 70% saturation. Gold accent `#D4A017` → dark mode equivalent `#C49512` (same hue, −30% sat).

---

### 2.1.3 CTA (Call-to-Action) Rules

1. **Maximum 1 primary CTA per screen/view.** If a screen has two equal actions, one must be secondary (ghost/outline style).
2. **CTA placement = thumb zone.** On mobile, primary CTA lives in the bottom 25% of the screen (reachable with one thumb). On desktop, bottom-right of modal or top-right of page header.
3. **Gold `#D4A017` = CTA color.** Only primary CTAs use gold. Secondary actions use navy outline. Destructive uses red `#DC2626`.
4. **CTA label describes the outcome**, not a generic verb.

| ❌ Avoid | ✅ Use instead |
|---|---|
| Submit | บันทึกข้อมูล (Save Info) |
| OK | เข้าใจแล้ว (Got it) |
| Click here | ดูรายงาน (View Report) |

---

### 2.1.4 Confirmation Dialog — No YES/NO Buttons

**Never use YES/NO or ใช่/ไม่ใช่.** Button labels must state the exact action being confirmed.

| ❌ Bad | ✅ Good |
|---|---|
| Are you sure? → Yes / No | ลบรายการนี้ไหม? → **ลบเลย** / **เก็บไว้** |
| Confirm? → Yes / No | ส่งข้อความ? → **ส่งเลย** / **แก้ไขก่อน** |
| Delete? → Yes / No | ลบออร์เดอร์? → **ลบออร์เดอร์** / **ยกเลิก** |
| Logout? → Yes / No | ออกจากระบบ? → **ออกจากระบบ** / **อยู่ต่อ** |

> **Rule:** The destructive action button must always be **red `#DC2626`** and placed on the **right** (or bottom on mobile). The safe/cancel action is always on the **left** (or top on mobile), in navy outline style.

---

### 2.1.5 Inbox Pagination & Load Strategy

Heavy conversation lists (Inbox) must **never** load all records at once — this blocks the main thread and increases server cost.

| Strategy | When to use | Implementation |
|---|---|---|
| **Cursor-based pagination** | Inbox conversation list | Load 20 items, "Load more" button at bottom, append to existing list |
| **Virtual scrolling** | Message thread (chat view) | Render only visible messages, recycle DOM nodes |
| **Skeleton loading** | Initial page load | Show 5–8 skeleton cards while fetching |
| **Optimistic updates** | Send message | Show message immediately, confirm/rollback on API response |

```
Inbox List: page_size = 20, cursor = last_conversation_id
Button: "โหลดเพิ่มเติม (Load more)" — NOT infinite scroll
Reason: Infinite scroll loses user's position on refresh; Load More preserves scroll position
```

> **NFR2 target:** Inbox list initial load < 500ms (Redis cached conversation list, Pusher for realtime updates)

## 5. Motion & Animation

### 5.1 Principles

1. **Purposeful:** Every animation must serve a UX purpose — guide attention, indicate state change, or provide feedback. No decorative motion.
2. **Fast:** Users should never wait for an animation. Keep durations short.
3. **Consistent:** Use the same easing and duration tokens across the app.

#### Duration Tokens

| Token | Duration | Usage |
|---|---|---|
| `duration-fast` | 100ms | Button press feedback, toggle |
| `duration-normal` | 200ms | Panel slide, card hover, dropdown |
| `duration-slow` | 300ms | Page transition, modal appear |
| `duration-slower` | 500ms | Skeleton shimmer cycle, AI sparkle |

#### Easing

| Token | Value | Usage |
|---|---|---|
| `ease-out` | `cubic-bezier(0.16, 1, 0.3, 1)` | Elements entering (slide in, fade in) |
| `ease-in` | `cubic-bezier(0.7, 0, 0.84, 0)` | Elements exiting (slide out, fade out) |
| `ease-in-out` | `cubic-bezier(0.45, 0, 0.55, 1)` | Layout shifts, repositioning |
| `spring` | `{ stiffness: 300, damping: 30 }` | Playful interactions (Framer Motion) |

### 5.2 Key Animations

| Element | Animation | Duration | Easing | Framer Motion |
|---|---|---|---|---|
| **New message** | Slide in from bottom + fade | 200ms | ease-out | `initial={{ y: 20, opacity: 0 }} animate={{ y: 0, opacity: 1 }}` |
| **Customer card** | Slide in from right | 250ms | ease-out | `initial={{ x: 360 }} animate={{ x: 0 }}` |
| **Dashboard cards** | Stagger fade in | 300ms total, 50ms stagger | ease-out | `variants` with `staggerChildren: 0.05` |
| **AI compose** | Shimmer gradient sweep | 1500ms loop | linear | CSS `background-position` animation on gradient |
| **Notification** | Scale up + subtle bounce | 300ms | spring | `initial={{ scale: 0 }} animate={{ scale: 1 }}` with spring |
| **Modal** | Fade backdrop + scale content | 200ms | ease-out | `AnimatePresence` + scale from 0.95 |
| **Page transition** | Fade + slight upward shift | 200ms | ease-out | `initial={{ opacity: 0, y: 8 }} animate={{ opacity: 1, y: 0 }}` |
| **Skeleton loading** | Shimmer pulse | 1500ms loop | ease-in-out | Tailwind `animate-pulse` or custom shimmer |
| **Button press** | Scale down briefly | 100ms | ease-in-out | `whileTap={{ scale: 0.97 }}` |
| **Toast notification** | Slide in from top-right | 200ms | ease-out | Auto-dismiss after 4000ms with fade out |

## 7. Accessibility

### 7.1 Color Contrast

| Context | Minimum Ratio | Standard |
|---|---|---|
| Body text on white | 7:1 | WCAG AAA |
| Heading text on white | 7:1 | WCAG AAA |
| UI text (labels, buttons) | 4.5:1 | WCAG AA |
| Large text (24px+) | 3:1 | WCAG AA |
| Non-text elements (icons, borders) | 3:1 | WCAG AA |

**Validation results for key combinations:**

| Foreground | Background | Ratio | Pass |
|---|---|---|---|
| Off-Black `#1A1A1A` | Cool White `#F5F8FB` | ~17:1 | AAA |
| Deep Navy `#101828` | Cool White `#F5F8FB` | ~18:1 | AAA |
| Mid Dark `#2D3748` | Cool White `#F5F8FB` | ~10:1 | AAA |
| Cool White `#F5F8FB` | Deep Navy `#101828` | ~18:1 | AAA |
| Deep Navy `#101828` | Saffron Gold `#D4A017` | ~8:1 | AAA |

**Note:** Cool white `#F5F8FB` on Deep Navy `#101828` passes AAA at all sizes. Deep Navy `#101828` on Saffron Gold `#D4A017` also passes AAA, making them the preferred CTA combination. For body text, always use Off-Black `#1A1A1A` on `#F5F8FB` background.

### 7.2 Touch Targets

- Minimum interactive target: **44px x 44px** (WCAG 2.2 Target Size)
- Spacing between adjacent targets: minimum **8px**
- Mobile form inputs: **44px height** minimum (our default is 44px)

### 7.3 Thai Screen Reader Considerations

- Use semantic HTML (`<nav>`, `<main>`, `<article>`, `<button>`) — not `div` with `onClick`
- All images and icons need `alt` text in Thai when meaningful, `aria-hidden="true"` when decorative
- Form labels must be associated with inputs via `htmlFor` / `id`
- ARIA labels for icon-only buttons: `<button aria-label="ส่งข้อความ"><Send /></button>`
- Language attribute: `<html lang="th">` with `lang="en"` spans for English text segments
- Page titles in Thai: `<title>กล่องข้อความ — Zuri</title>`

### 7.4 Keyboard Navigation

- All interactive elements reachable via Tab
- Focus ring: 2px solid `#D4A017`, 2px offset (visible on both light and dark backgrounds — Saffron Gold contrasts with both deep navy and cool white)
- Escape closes modals, dropdowns, side panels
- Enter/Space activates buttons and toggles
- Arrow keys navigate within lists, menus, and grids
- Skip-to-content link as first focusable element

### 7.5 Reduced Motion

Respect `prefers-reduced-motion: reduce`:
- Replace all transitions with instant state changes
- Disable parallax and decorative animations
- Keep essential state-change indicators (e.g., loading spinners) but simplify them

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```
